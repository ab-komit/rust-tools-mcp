# Create a Tool Plugin for rust-tools-mcp

This guide explains how to write, build, and install Rust `cdylib` plugins for the `rust-tools-mcp` MCP server. Plugins are shared libraries (`.so`/`.dylib`/`.dll`) loaded at runtime via `libloading`. Each plugin can expose one or more MCP tools with typed JSON Schema inputs.

## How the MCP server works

The host (`mcp-host`) is a JSON-RPC 2.0 server that communicates over stdin/stdout. It loads plugins from two locations:

| Scope | Path |
|---|---|
| Global | `~/.config/rust-tools/` |
| Project | `<project-root>/.rust-tools/` |

Project tools override global tools with the same name.

### Plugin ABI

Every plugin `.so` exports these 5 C symbols (auto-generated by the `#[tool_plugin]` proc macro):

| Symbol | Purpose |
|---|---|
| `plugin_abi_version` | Returns `ABI_VERSION` (currently 1) for compatibility |
| `plugin_name` | Returns the module name as a C string |
| `plugin_list_tools` | Returns a JSON array of `ToolDescriptor` objects (name, description, inputSchema) |
| `plugin_call_tool` | Takes tool name + args JSON, returns `ToolResult` JSON |
| `plugin_free_string` | Frees strings allocated by the plugin |

The server receives `tools/list` and `tools/call` JSON-RPC requests and dispatches them to the appropriate plugin via these C FFI symbols.

### MCP protocol

- Protocol version: `2024-11-05`
- Transport: JSON-RPC 2.0 over stdin/stdout
- Methods handled: `initialize`, `tools/list`, `tools/call`

## Prerequisites

- Rust toolchain (edition 2021)
- The `rust-tools-mcp` repo cloned (for the SDK crates)

## Writing a plugin

### 1. Create a new crate

`Cargo.toml`:

```toml
[package]
name = "my-tool"
version = "0.1.0"
edition = "2021"

[lib]
crate-type = ["cdylib"]

[dependencies]
mcp-plugin-sdk = { path = "/path/to/rust-tools-mcp/crates/mcp-plugin-sdk" }
mcp-plugin-sdk-macros = { path = "/path/to/rust-tools-mcp/crates/mcp-plugin-sdk-macros" }
serde = { version = "1", features = ["derive"] }
schemars = "0.8"
```

Or if the plugin lives inside the `rust-tools-mcp` workspace, use workspace-relative paths:

```toml
mcp-plugin-sdk = { path = "../../crates/mcp-plugin-sdk" }
mcp-plugin-sdk-macros = { path = "../../crates/mcp-plugin-sdk-macros" }
```

### 2. Define args and tool functions

`src/lib.rs`:

```rust
use mcp_plugin_sdk::serde::Deserialize;
use mcp_plugin_sdk::schemars::JsonSchema;
use mcp_plugin_sdk_macros::{tool, tool_plugin};

// Args struct — must derive Deserialize + JsonSchema
#[derive(Deserialize, JsonSchema)]
struct ReviewArgs {
    /// Path to the file to review
    file_path: String,
    /// Whether to run security checks
    #[schemars(default)]
    check_security: Option<bool>,
}

// Wrap tools in a #[tool_plugin] module — generates the ABI symbols
#[tool_plugin]
mod tools {
    use super::ReviewArgs;

    /// Reviews code for security issues — this doc comment becomes the tool's description
    #[tool]
    fn code_review(args: ReviewArgs) -> Result<String, String> {
        Ok(format!("Reviewed {} (security={})", args.file_path, args.check_security.unwrap_or(false)))
    }
}
```

### Conventions

- The args struct **must** derive both `Deserialize` and `JsonSchema`
- Use `#[schemars(default)]` on `Option<T>` fields to make them optional in the JSON Schema
- Doc comments (`///`) on struct fields become field descriptions in the schema
- The doc comment on the `#[tool]` function becomes the tool's MCP description
- Function signature must be: `fn name(args: YourArgs) -> Result<String, String>`
- Multiple `#[tool]` functions can live in the same `#[tool_plugin]` module
- The module name (e.g., `tools`) becomes the plugin name returned by `plugin_name()`

## Building and installing

```bash
# Build the MCP host (once)
cargo build --release -p mcp-host

# Build your plugin
cargo build --release -p my-tool

# Install globally (available to all projects)
mkdir -p ~/.config/rust-tools
cp target/release/libmy_tool.so ~/.config/rust-tools/

# Or install per-project (overrides global on name collision)
mkdir -p .rust-tools
cp target/release/libmy_tool.so .rust-tools/
```

## Configuring an MCP client

### opencode

Add to `opencode.json` or `opencode.jsonc`:

```jsonc
{
  "mcp": {
    "servers": {
      "rust-tools": {
        "command": "/path/to/rust-tools-mcp/target/release/mcp-host"
      }
    }
  }
}
```

Restart opencode after installing new plugins — tools are loaded at startup only.

### Other MCP clients

The server speaks standard MCP over stdio. Point your client's MCP server configuration to the `mcp-host` binary.

## Verification

Test the server directly:

```bash
# List tools
echo '{"jsonrpc":"2.0","id":1,"method":"tools/list"}' | /path/to/mcp-host

# Call a tool (replace "greet" with your tool name, adjust args)
echo '{"jsonrpc":"2.0","id":2,"method":"tools/call","params":{"name":"greet","arguments":{"name":"World"}}}' | /path/to/mcp-host
```

Enable debug logging:

```bash
RUST_LOG=debug /path/to/mcp-host
```

## Reference: key types

```rust
// mcp_plugin_types::ToolDescriptor
pub struct ToolDescriptor {
    pub name: String,
    pub description: String,
    pub input_schema: serde_json::Value, // JSON Schema
}

// mcp_plugin_types::ToolResult
pub struct ToolResult {
    pub content: Vec<ContentItem>,
    pub is_error: bool,
}

// mcp_plugin_types::ContentItem
pub struct ContentItem {
    pub type_: String,   // "text"
    pub text: String,
}

// Helper — construct results
ToolResult::success("ok")
ToolResult::error("something went wrong")
```
